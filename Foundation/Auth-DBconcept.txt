4. Password Hashing: 
- Password Hashing is a way to store password in a secure way
//suppose pass: Beauty123

json: {"password":"Beauty123"} what if database leak everyone sees the password in plain text

so we use password hashing 
ex: Beauty123 -> Hash function -> ajf8j3k4f8sdf89sdf
store: {"password":"ajf8j3k4f8sdf89sdf"}

//login flow: user enter pass -> system hashes again and then compare the new hash value with 
db hash value if matches then login succes otherwise failed

//why can't reverse the hash value?
//hash value is random so can't reverse it and it is one way only can't go back to original value
// ex: bcrypt argon2

//note : never use MD5 SHA1 for password becuase it is too fast attackers can brute-force it means brute-force attack (brute force means try all possible value)
// use bcrypt or argon2 for password hashing bcoz they are intentionally slow


//why not encryt password:
//we never need the original password again. we only need verification so hashing is safer than encryption


5.JWT token concept:
- JSON Web Token (JWT) is an open standard (RFC 7519) for securely transmitting information between parties as JSON object
- this is the important part of modern authentication
- tradition session:
 user login -> server stores session -> session id sent to browser (server must remember everyone)

 -> JWT Approach:
 -> user login -> server creates JWT token -> JWT token sent to browser or client -> it stores token Authorization: Bearer TOKEN 
 -> server verify token no db loopup required in this (stateless)

//note: JWT is stateless means we don't need to store the user detail in the server or database

//jsw structure: Header Payload Signature like xxxx.yyyy.zzzz

.typical flow:
Login
 ↓
Verify Password
 ↓
Generate JWT
 ↓
Send Token
 ↓
Client Stores Token
 ↓
Request With Token
 ↓
Verify JWT
 ↓
Access Granted


//jwt cannot be easily revoked:
//ex: user logs out token may still remain valid 
//thats why many companies use access token -> refresh token instead of single jwt

//how login works using hashing and JWT:
1. User enters email and password.

2. Backend finds user in database.

3. Password is hashed and compared with stored bcrypt hash.

4. If matched, server generates JWT containing user id and role.

5. JWT is sent to client.

6. Client sends JWT in Authorization header on future requests.

7. Backend verifies JWT signature and extracts user information.

8. Authorization middleware checks role/permissions before allowing access.
